什么是Git

Git 是目前最流行的分布式版本控制系统，由 Linux 之父 Linus Torvalds 在 2005 年开发。它就像一个项目的"超级存档"，能精准记录每个文件的每次改动，并让多人协作高效协同。

被 Git 管理的文件夹，会在根目录生成隐藏子文件夹 .git，存放所有版本控制信息。

Git有两种仓库类型，分为本地仓库和远端仓库。

  • 本地仓库：即本地受 Git 管理的项目目录，版本记录存放在项目根目录的.git文件夹下。
  • 远端仓库：将本地仓库上传到远端托管平台，如 GitHub、Gitee 等。一般分为公开(public)仓库和私有(private)仓库，公开仓库所有人都能够访问与下载，私有仓库只有仓库所有者及被授权的协作者能够访问与下载。

所以Git 是工具本身，用于在本地进行版本控制。GitHub（以及 GitLab、Gitee）是远端托管平台，两者是完全不同的概念。

如何使用

要使用Git首先要安装Git，Windows可以在Git官网下载。

基本概念

如何开始让Git管理文件？

在需要被管理的目录下打开命令行窗口，在其中输入git init，然后在当前目录下就会生成一个隐藏的 .git 子文件夹，删除这个文件夹就可以取消 Git 管理

有不想被管理的文件怎么办？

若有不希望被管理的文件，可以创建一个.gitignore文件，在其中指定需要忽略的文件或目录，比如包含敏感信息的文件、项目依赖文件等。

注意，.gitignore 只能忽略未被跟踪的文件，对已被 Git 跟踪的文件无效，需先 git rm --cached 移除跟踪，所以建议在最初就写好需要忽略的文件。

版本控制

Git不会时刻自动保存你的修改，每个存档点需要手动提交(commit)。

以IDEA的图像界面为例：

点击提交后，会保存仓库当前状态的快照，并记录提交信息和生成一个唯一的Commit ID。

同样可以在IDEA中直观地看见提交历史。

若需要撤销已提交的commit，有以下三种策略：

  1. Discard：放弃尚未 Commit 的本地修改。
  2. Reset：将仓库强制回退到某个历史 Commit，适用于单人分支或未推送远端的场景。若已推送远端，则需要 --force 强制推送，存在覆盖他人代码的风险，多人协作禁止使用。
  3. Revert：生成一个反向 Commit 来抵消某次 Commit 的改动，安全且适合多人协作，推荐在多人协作场景下使用。

分支

分支（Branch） 是你项目的一条独立开发线。可以把 Git 分支想象成一个 可移动的指针，指向某次提交的"快照"。当你新建分支时，Git 只是创建了一个新指针，并不会拷贝整个项目，这也是它速度极快的原因。

默认分支名通常为main，早期通常为master。可以在任意一个历史提交创建一个新分支。

HEAD就是一个特殊的指针，表示你当前所在的分支。切换分支，其实就是移动 HEAD。

不同分支上新产生的提交不会自动出现在其他分支上，比如创建了一个新分支后，在新分支的新的提交记录不会出现在原分支上（创建分支前的提交记录会被新分支共享）。

可以把分支的改动并入另一分支，合并后通常会删除已完成使命的分支，注意不能删除当前所处的分支，需先切换到其他分支再删除。

问题来了，两个分支的内容相互冲突怎么办，比如一开始有段内容是"吃饭"，主分支将其改为了"睡觉"，新分支将其改为了"喝水"，那合并的时候应该听谁的呢？

当两个分支修改了同一个文件的同一行代码时，合并时 Git 无法自动确认该保留哪个改动，就会产生合并冲突。若修改的是同一文件的不同行，Git 则可以自动合并。冲突需要人工解决，通常有三种选择：

  1. 保留 A 分支的修改，丢弃 B 分支。
  2. 保留 B 分支的修改，丢弃 A 分支。
  3. 两个都保留，手动编辑成最终想要的内容。

如果只想将部分提交合并入主干，可以使用 cherry-pick（拣选提交）：将某个或多个 Commit 的改动单独应用到当前分支，而不合并整个分支。

可以复制想要拣选的提交 ID，然后使用 git cherry-pick <commit-id> 命令将其提交到当前分支。

如果遇到问题需要马上切换到别的分支，但当前分支还有未完成的代码，不想让这些代码并入到别的分支，这时候可以利用 git stash（暂存工作现场）临时保存一下，到时候切换回来再取出来即可。

还有一种操作，叫做 rebase（变基），作用是将一个分支的提交“嫁接”到另一个分支的最新提交之后，使提交历史保持线性，避免产生额外的 Merge 提交。比如将 temp 变基到 main 上后，temp 的提交会被移至 main 最新的提交之后，原来的 temp 分支依然存在，只是其提交历史被重写了，不会再产生额外的 Merge 提交。

需要注意的是，如果该分支之前已推送到远端，变基后必须使用 强制推送 (git push --force 或 --force-with-lease) 来更新远端分支。严禁在多人共享的分支上执行变基，否则会覆盖他人代码，导致严重协作问题。仅适用于自己一个人使用的分支。

四个关键区域

Git 涉及四个区域：

  1. 工作目录（Working Directory）：本地磁盘上的项目文件夹。
  2. 暂存区（Staging Area/Index）：Commit 前的检查点，用于精细控制哪些修改被提交。传统流程先 git add 将文件加入暂存区，再 git commit 提交暂存区中的所有内容。
  3. 本地仓库（Local Repository）：通过 Commit 将暂存区的改动存入本地版本库。
  4. 远端仓库（Remote Repository）：托管在 GitHub 等服务器上的仓库。

最常用的操作命令：

  • git clone：把远端仓库克隆到本地，同时创建本地仓库和工作目录。
  • git add + git commit：工作区 → 暂存区 → 本地仓库。
  • git push：本地仓库 → 远端仓库。
  • git pull = git fetch + git merge：将远端改动拉取并合并到本地分支，同时更新工作目录。

Github核心功能

绑定远端仓库通常有两种方式：

  • 本地已有仓库，关联到远端空仓库
  • 克隆远端仓库，自动完成绑定

多人协作时，通常先 pull 远端最新改动，然后再继续开发。

GitHub 仓库有以下核心模块：

  • 代码区 (Code)：显示项目全部源代码，可按文件名搜索（快捷键 T），点击文件可查看内容。
  • 提交信息与日期：每个文件旁边显示最后一次 Commit 的说明和更新时间。
  • README.md：自动渲染的项目说明文档，介绍项目用途、使用方法等。
  • Release：项目发布版本，通常提供可直接下载的软件包。
  • Star：类似收藏，反映项目热度，可以在个人主页找到自己点过Star的仓库。
  • Fork：将项目复制一份到自己账号下，便于学习或贡献。
  • Issues：用于报告 Bug、提出新功能或讨论问题，分为 Open 和 Closed 两种状态。
  • Pull Request (合并请求)：向项目贡献代码的方式。

GitHub 仓库页面还可以使用一些快捷键：

  • 按 /：快速聚焦搜索栏。
  • 按 T：文件查找。
  • 按 L：跳转到指定行号。
  • 按 ?：查看所有快捷键列表。
  • 按 .：在浏览器中打开网页版 VS Code 在线编辑环境。

贡献代码

为开源项目贡献代码的流程如下：

  1. Fork：将目标开源仓库 fork 到自己的 GitHub 账号下，得到一份副本。

  2. Clone：将自己账号下的仓库克隆到本地。

  3. 同步主线并创建分支：先将上游（源项目）的最新改动拉取到本地，确保自己仓库的 main 分支与上游同步，然后基于 main 分支创建新的功能分支。严禁直接在主分支上修改。

  4. 开发与提交：在新分支上进行代码修改，完成 Commit 并 Push 到自己的远端仓库。

  5. 同步上游：在发起 PR 前，先将上游的最新改动同步合并到当前分支，并在本地解决可能产生的冲突。

  6. 创建 Pull Request (PR)：

    在 GitHub 上进入自己 Fork 的仓库，点击"Pull Request"，确保合并方向是从自己仓库的分支到源项目的 main 分支，填写标题和描述后提交。

  7. Code Review 与合并：

    源项目的维护者收到 PR 通知，可以在"Files changed"中审查代码，提出修改意见或直接批准。维护者点击"Merge pull request"并 confirm，贡献的代码就会合并进上游主干。

若是在自己团队内部合作开发就无需 fork 代码了，项目管理员在仓库设置中的"Collaborators"添加团队成员，成员被授权后可直接 Push 到仓库。成员直接创建分支进行开发，推送后仍通过 PR 方式进行代码审查与合并，以保证代码质量。

多人开发中的关键原则就是：

  • 始终在分支上开发，保持主线干净。
  • 发起 PR 前必须先同步上游最新代码并在本地解决冲突，避免 PR 被阻塞。
